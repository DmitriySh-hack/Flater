# Глава 1. Предпроектный анализ

В данной главе рассматриваются ключевые аспекты, связанные с созданием веб-приложения для размещения и поиска объявлений о продаже и аренде жилой недвижимости (проект **FlatSell**), оцениваются тенденции рынка, формулируются требования к системе, выполняется сравнительный анализ аналогов и обосновывается выбор технологического стека, реализованного в репозитории приложения.

## Веб-платформы в сфере недвижимости и объявлений о жилье

Онлайн-сервисы по продаже и аренде квартир объединяют каталог объектов, средства коммуникации между продавцом и покупателем и, при необходимости, внутренние инструменты для сотрудников агентства (CRM). В отличие от классической «витрины» интернет-магазина, предметная область опирается на уникальные объекты (квартиры), географию, сроки просмотра и договорённости о бронировании, а не на однотипную корзину товаров со складскими остатками.

**Преимущества** таких систем для пользователя:

- доступ к объявлениям без визита в офис и в удобное время;
- фильтрация по городу, цене и другим параметрам;
- сохранение понравившихся объектов и планирование просмотров;
- обмен сообщениями с контрагентом по объекту.

**Ограничения и риски**:

- высокая конкуренция у крупных агрегаторов;
- необходимость защиты персональных данных и учётных записей;
- поддержка актуальности объявлений и медиафайлов (фотографии жилья).

**Вывод по разделу.** Анализ предметной области показывает целесообразность разработки специализированного веб-приложения с клиентской частью на React, серверным API на Node.js (Express), хранением данных в Microsoft SQL Server и дополнительными функциями бронирования, календаря, избранного, переписки и отдельного модуля CRM для сотрудников — что соответствует реализованной в проекте FlatSell функциональности.

## 1.1. Описание предмета и объекта исследования

Современный рынок недвижимости опирается на цифровые каналы: потребители ищут квартиры через сайты и приложения, сравнивают цены и локации, оставляют заявки и договариваются о просмотрах онлайн. Для агентств и площадок важны не только публикация карточек объявлений, но и учёт обращений, роли сотрудников и устойчивость работы сервиса под нагрузкой.

В контексте разработки информационной системы **FlatSell** значимы понятия: пользователь и профиль, объявление о квартире (медиа, атрибуты объекта), избранное, бронирование и календарь слотов, диалоги (в т.ч. в режиме реального времени через WebSocket на сервере), а также **сотрудник CRM** с должностью (например, super-admin, manager, support) и отдельной аутентификацией.

**Объект исследования** — деятельность по представлению и сопровождению объявлений о жилой недвижимости в сети Интернет (поиск, размещение, взаимодействие сторон сделки и внутренняя работа персонала).

**Предмет исследования** — проектирование и программная реализация веб-приложения для размещения объявлений о квартирах, включая автоматизацию связанных процессов (учёт пользователей и объявлений, бронирование по календарю, обмен сообщениями, CRM-сотрудники), с использованием выбранного стека технологий (React, TypeScript, Vite, Express, MS SQL Server, MobX и др.), отражённого в структуре проекта `flatSell`.

## 1.2. Постановка задачи на проектирование

Исходя из реализованных в проекте модулей (`components`, `backend`, `CRM`), к разрабатываемой системе предъявляются следующие группы требований.

**Пользователь и безопасность**

- регистрация, вход, выход, обновление сессии (JWT в cookie, эндпоинты в духе `/registration`, `/login`, `/logout`, `/refresh`);
- активация учётной записи по ссылке, смена пароля, обновление профиля;
- загрузка аватара с ограничением типа и размера файла (multer, память/диск).

**Объявления о недвижимости**

- создание, изменение, удаление объявления авторизованным пользователем;
- загрузка изображений (ограничение числа и размера файлов);
- выдача списков объявлений, привязка к пользователю, публичные выборки (в т.ч. все объявления, города);
- получение объявления с данными пользователя для карточки.

**Взаимодействие пользователей**

- избранное: добавление, просмотр, удаление;
- бронирование: создание, отмена, получение списка;
- календарь: просмотр занятости по объявлению, резервирование, удаление записей (в т.ч. по идентификатору записи);
- сообщения: история переписки, список диалогов, отметка прочитанного; серверное WebSocket-подключение для оперативной доставки.

**CRM для сотрудников**

- отдельный входовой контур (`/crmsys` на клиенте, маршруты `/employee/*` на API);
- регистрация сотрудника, вход, выход, refresh-токен;
- справочник должностей (`/employee/positions`, фиксированный перечень позиций в коде бэкенда);
- смена пароля, выдача списка сотрудников для административных задач.

**Нефункциональные требования**

- разделение клиентского URL и API (`/api`), CORS с учётом credentials;
- централизованная обработка ошибок на сервере;
- хранение данных в реляционной СУБД Microsoft SQL Server (пул соединений `mssql` / драйвер `msnodesqlv8` в конфигурации проекта);
- удобство сопровождения: TypeScript на фронтенде, модульная структура бэкенда по доменам (advertisement, booking, calendar, messages, employeeCRM и т.д.).

Эти требования задают границы первой очереди функциональности и служат основой для главы проектирования (модели данных, интерфейсы, тестирование).

## 1.3. Анализ потенциальных конкурентов

Для ориентира при проектировании пользовательского опыта и набора функций целесообразно сопоставить разрабатываемую систему с распространёнными российскими сервисами объявлений и недвижимости.

**Примеры для сравнения**

1. **ЦИАН** (cian.ru) — крупный агрегатор объявлений, карты, фильтры, мобильные приложения.
2. **Авито: Недвижимость** (avito.ru) — универсальная доска объявлений с сильным трафиком и простым размещением.

**Критерии оценки** (каждый по шкале 1–5, как в примере ВКР):

| Характеристика / сервис        | ЦИАН | Авито (недвижимость) |
|-------------------------------|------|------------------------|
| Глубина фильтрации и карты    | 5    | 4                      |
| Простота размещения объявления | 4 | 5                      |
| Личные сообщения / чаты       | 5    | 5                      |
| Календарь просмотров у частного объявления | 3 | 2           |
| Встроенная CRM для сотрудников агентства (в одном продукте с «витриной») | 2 | 2 |

**Вывод.** Крупные площадки сильны в охвате аудитории и маркетинге; учебный/кастомный проект **FlatSell** может конкурировать не масштабом, а **прозрачностью кода**, полным контролем сценариев (календарь бронирования, CRM с ролями), возможностью адаптации под конкретное агентство или учебный кейс и использованием современного стека (SPA на React + отдельный REST/WebSocket backend).

## 1.4. Выбор технологий и обоснование их использования

Ниже перечислены технологии, фактически применённые в репозитории `flatSell` (см. `package.json`, `vite.config.ts`, `backend/server.cjs`, `backend/db-mssql.cjs`), и причины их выбора.

### React и TypeScript

**React** — декларативная библиотека для построения пользовательского интерфейса с компонентной архитектурой, удобной для страниц каталога, профиля, форм объявлений и изолированного CRM-интерфейса.

**TypeScript** повышает надёжность клиентского кода за счёт статической типизации (модели в `components/models`, сервисы API), что снижает количество ошибок при рефакторинге.

### Vite

**Vite** — современная среда сборки и dev-сервер для фронтенда: быстрый старт, горячая замена модулей (HMR), интеграция с React через официальный плагин. Это ускоряет разработку по сравнению с тяжёлыми конфигурациями legacy-сборщиков.

### React Router

**react-router-dom** обеспечивает маршрутизацию SPA: отдельные маршруты для домашней страницы, профиля, избранного, бронирований, сообщений, а также **отдельная ветка `/crmsys`** для CRM без общего header/footer основного приложения (`App.tsx`, `CRMRoute.tsx`).

### MobX

**Mobx** и **mobx-react-lite** используются для реактивного состояния (например, `components/store/store.ts`): удобно связать авторизацию, загрузку объявлений и UI без избыточного шаблонного кода.

### Axios

**Axios** — HTTP-клиент для вызовов REST API с перехватчиками и единообразной работой с JSON; типичен для связи React-приложения с Express-бэкендом.

### Node.js и Express

**Express** (версия 5.x в зависимостях) — минималистичный фреймворк для маршрутизации и middleware на Node.js: маршруты сгруппированы в `backend/router/index.cjs`, подключаются контроллеры доменов, **express-validator** для проверки входных данных, **multer** для multipart-загрузок изображений.

### Microsoft SQL Server

В проекте используется **MS SQL Server** (конфигурация пула в `db-mssql.cjs`, база `flater_db`, драйвер **msnodesqlv8**). Обоснование: интеграция с инфраструктурой Windows/SQL Server в учебных и корпоративных средах, транзакционная модель, знакомый администраторам инструментарий.

### Аутентификация и безопасность

**jsonwebtoken**, **bcrypt** / **bcryptjs**, **cookie-parser** — реализация выдачи и обновления токенов, хэширование паролей, работа с httpOnly-cookie на стороне API.

### Дополнительно

- **ws** — WebSocket-сервер поверх HTTP для сообщений (`messages/websocket.cjs`).
- **cors** с `credentials: true` и `origin` из переменных окружения — безопасное взаимодействие браузера с API.
- **dotenv** — вынесение секретов и URL клиента из кода.
- **nodemailer** — отправка писем (например, активация).
- **cloudinary** (в зависимостях) — потенциал выноса медиа в облако при развитии проекта.
- **uuid** — генерация уникальных идентификаторов при необходимости.

### Среда разработки

Для редактирования кода и отладки удобно использовать **Visual Studio Code** (или аналог) с поддержкой TypeScript, ESLint и встроенной работы с Git.

### Обобщающее обоснование

Выбор стека **React + TypeScript + Vite** на клиенте и **Express + MS SQL Server** на сервере позволяет:

- чётко разделить ответственность фронтенда и API;
- масштабировать функциональность по модулям (объявления, бронирование, календарь, сообщения, CRM);
- использовать единый язык JavaScript/TypeScript в экосистеме Node.js;
- опираться на зрелые библиотеки с активным сообществом и документацией.
